OAuth 2.0 简介

在 2010 年以前，大多数的身份认证场景都在想办法解决一个问题，如何让用户在不输入密码的前提下授权第三方访问用户的数据？例如使用 Google，Facebook，微信帐号授权登录操作，这就是 OAuth 诞生的背景。

OAuth 2.0 用于委托授权的案例

假设用户 A 有一个 Google 帐号，A 信任 Google。现在 A 想要访问 Yelp 网站，Yelp 网站支持使用 Google 帐号进行授权登录，当 A 点击某个 Connect to Google 按钮开始，OAuth Flow 便开始了:

  1. 用户 A 点击 Connect to Google 按钮
  2. 重定向至 Google Authorization Service，该服务让用户 A 首先验证其在 Google 的身份，体现为帐号密码输入框，用户 A 输入用户密码是向提供 Google 提供了机密信息而不是 Yelp。
  3. 成功登录后，Google Authorization Service 将询问用户 A: Yelp 想要访问他的 Google 个人资料(Profile)和联系清单(Contacts)，是否同意授权？
  4. 若用户 A 选择否，授权中止；若用户 A 选择同意，Google 会将用户重定向至之前的登录页面并携带一个提供给 Yelp 用于向 Google 索取用户 A 相关信息的令牌，Yelp 随后使用该令牌向 Google 服务器发送请求，得到用户 A 的个人信息。

OAuth 2.0 术语对照

  • Resource Owner(用户 A，数据的主人)
  • Client(Yelp，想要获得访问用户 A 数据授权的机构)
  • Authorization Server(授权服务器，形如 accounts.google.com)
  • Resource Server(资源服务器，用于取得用户 A 资料的 Google Profile API 或 Google Contacts API)
  • Authorization Grant(授权行为，用户 A 的授权处理过程)
  • Redirect URI(Authorization Server 需要知道用户同意授权后的回调地址)
  • Access Token(Client 最终用来向 Resource Server 获取用户数据的凭证)

委托授权的实现流程

Authorization Code 模式

  1. 首先，Google 支持用户在访问 Yelp 时可以使用 Google 账号进行授权登录，同时谷歌向 Yelp 发放 API Secret。
  2. Yelp 将 Connect to Google 按钮指向的 url 按照 Google 规定的格式填写，并在该 url 中以查询字符串参数带上回调的 url，scope，response type 及其他配置信息，形如

    \https://www.google.com/auth?url=www.yelp.com/authorize&scope=profile&response_type=code

  3. 用户 A 在 Google Authorization Service 返回的页面输入帐号密码成功登录并同意授权
  4. Google Authorization Service 向浏览器返回一个重定向响应，该重定向 url 即 Yelp 指定的 url 并以查询字符串参数带上 code 参数。
  5. 该 url 由 Yelp 指定并指向 Yelp 的某个服务端 Api，该 Api 使用该 code 并结合从 Google 获取的 API Secret 再次访问 Google Authorization Service 换取一个 access token。
  6. 在得到 access token 之后，Yelp 服务端使用该 token 向 Google Resource Server(Profile API) 请求用户 A 的数据。

此处 Yelp 仅仅想要读取 Profile 信息，但它不能删除 Contacts 信息，所以读取和修改 Profile 就是不同的 Scope。Yelp 作为客户端请求授权时需要提供它想要获得授权的 Scope，以让 Google Authorization Service 在询问用户时向其展示 Yelp 想要访问哪些数据。用户同意授权之后 Google Authorization Service 返回给 Yelp 的 code 最终换取的 access token 也就被限制在这些 Scope 中。

Q&A: 为何不直接返回 access token 而是首先返回 code，再去交换 access token？

Back Channel 和 Front Channel

  • Back Channel: 由系统的后台服务向另一个系统的 API 发起的请求，被认为是 Back Channel，由于开发人员对后台程序代码具有完全的控制权，且外部人员无法轻易取得访问服务器的权限，所以这种方式具有更高的安全性。
  • Front Channel: 浏览器，由于浏览器是一个开放的代码执行平台，任何人都可以通过 View Source 查看源代码，其主要数据和信息都用于展示，所以比起 Back Channel，敏感数据不应该走 Front Channel。

浏览器向 Google 发起的授权请求的所有查询参数和 url 都是公开透明的，而 Google Authorization Service 返回的也是一个包含 code 查询参数的 url，它最终由浏览器执行重定向操作。如果此时有人拦截了该 code，并试图先于 Yelp 换取 access token 就会发生数据泄漏。但实际情况是，浏览器在获取到包含 code 的 url 后，该 url 指向了 Yelp 的后台 Api，该 Api 再使用该 code 并结合存储在服务端后台的 API Secret 一起去换取 access token。即便有人拦截了 code 的值，在不知道这个 Secret 的前提下，是无法换取 access token 的，所以换取 token 这一步操作走 Back Channel 提高了安全性。

OAuth 2.0 Grant Type

  • Authorization Code(front channel + back channel): 上述讨论的场景，首先返回 code，再以 code 去后台交换 access token
  • Implicit(front channel only): 单页应用的场景，无后台服务端，与 Authorization Code 模式的区别在于，直接返回 token，不走 Back Channel，这种方式有 token 被盗的风险，客户端 app 要确保 token 的安全。
  • Resource owner password credentials(back channel only): 使用用户凭证直接访问资源服务器，不推荐
  • Client credentials(back channel only): 常见于 machine to machine 或 service to service 的场景，即服务端保存了用户的认证凭证。
